# 风格镜像副驾（Style Mirror Copilot）

> Idea-refine 阶段产出。下一步：`/agent-skills:spec` 扩展为完整规格说明书。

## Problem Statement

**HMW** 让用户在社交软件聊天时，借助 AI 实时获得"高度匹配自己真实聊天风格"的候选回复，从而以最贴近真我的方式表达自己，筛选真正喜欢他本人风格的人？

## Recommended Direction

做一个安卓 App，从架构层面把"输入方式 / LLM 提供商 / 目标平台"都做成可扩展的接口，MVP 阶段先实现优先级最高的一组，后续按 P1/P2 增量交付。

**两条相互独立的数据流水线**：

```
【批量导入流水线】（一次性 / 周期性，重）
微信导出 / 微信备份 / 批量截图 / 纯文本 / 第三方工具产物
        ↓
数据清洗（去重、合并、过滤系统消息）
        ↓
说话人对齐（大规模识别"我" vs "别人"）
        ↓
采样与聚合（按对象/时间/话题分桶，避免偏置）
        ↓
性格画像提取（多维度结构化画像）
        ↓
┌────────────────────────────────────┐
│   StyleEngine（本地风格指纹存储）    │
└────────────────────────────────────┘
        ↑
增量学习（轻）：日常采纳/修改/丢弃信号 → 微调画像
        ↑
【日常使用流水线】（实时，轻）
用户输入（截图/复制粘贴/悬浮窗/分享）
        ↓
ConversationContext（统一上下文抽象：说话人 + 消息序列）
        ↓
StyleEngine 匹配 → LLMProvider 生成
        ↓
3 候选展示 + 反馈环
```

**两条独立性原则**：

1. **导入数据源 ⊥ 目标聊天平台**
   Onboarding 阶段塞进来的可能是几年的微信记录（数据最丰富），但 MVP 日常使用是在 Soul 上给候选。微信作为导入数据源不要求微信进入 P0 目标平台。

2. **批量学习 ⊥ 增量学习**
   批量学习从历史海量数据提炼稳定风格画像（精度下限的保证）；增量学习从日常使用信号微调（持续校准）。两条线写入同一个风格指纹存储。

**关键判断**：

- **LLM 默认 DeepSeek，保留 Claude 切换接口**
  DeepSeek 中文聊天语料训练得更好；价格约 Claude 的 1/30；数据不出境。Prompt 框架做成 provider-agnostic，效果不行可随时切 Claude / Qwen。

- **P0 同时支持截图 + 复制粘贴**
  共用一条 ConversationContext 管线，多写一个 input adapter 成本极低，但用户灵活度大很多。

- **悬浮窗、微信适配放 P1**
  PlatformAdapter / InputAdapter 抽象出来后，按需启动；不在 P0 但也不延后到 P3。

自用先行，但架构不绑死自用。

## MVP Scope（P0）

### 批量导入引擎（onboarding 与离线再训练共用）

| 模块 | 描述 |
|---|---|
| M10 批量导入入口 | 多源支持：① 微信 PC 端导出（最丰富）② 微信安卓备份 ③ 批量截图（数百张图，OCR 流水线复用 M1）④ 纯文本（用户自整理）⑤ 第三方工具产物（如 WeChatMsg 解析结果） |
| M11 数据清洗 | 去重、合并连续消息、过滤系统消息 / 转账 / 链接卡片 / 红包提示等噪声 |
| M12 说话人对齐（大规模） | 多对象、多设备、昵称变更、群聊场景下稳定识别"我" vs "别人"，错位率目标 < 2% |
| M13 采样与聚合策略 | 上万条消息不能直接喂 LLM。按对象 / 时间段 / 话题分桶采样，保留多样性，避免被某一段密集对话主导画像 |
| M14 性格画像提取 | 输出多维度结构化画像：语言风格、情感表达、幽默类型、回避模式、节奏特征、敏感话题处理。这是"风格指纹 v1"的真正起点 |
| M15 隐私边界（强制） | 原始数据全本地存储；仅"我"发出的消息聚合到画像；对方消息仅作为对话上下文情境标注，不进入画像聚合，不上传 LLM API（除日常使用时的最小必要上下文） |

### 日常使用流水线

| 模块 | 描述 |
|---|---|
| M1 输入抽象层 + 截图导入实现（OCR） | 复用同一 OCR 管线服务批量导入 |
| M2 输入抽象层 + 复制粘贴实现 | 单条 / 多条都支持 |
| M3 说话人识别（实时） | Soul 适配器，复用 M12 的核心算法 |
| M4 风格指纹存储与匹配 | 结构化 schema + few-shot 样本池，本地存储；提供查询接口供候选生成调用 |
| M5 LLM 抽象接口 | DeepSeek 实现 + Claude 切换接口 |
| M6 候选生成 | 默认 3 候选，可配置 1+refresh；每条标注风格匹配度 |
| M7 一键复制候选回复 | 切回 Soul 粘贴发送 |
| M8 反馈循环 | 采纳 / 修改 / 丢弃 → 增量学习写回 StyleEngine |

### Onboarding 串联

`首次启动 → 引导用户选择导入源 → 跑批量导入引擎 → 生成画像 v1 → 进入日常使用`

支持后续追加导入（新数据进来 → 重跑批量流水线 → 画像版本升级）。

## Phase 2（P1，P0 跑通后增量交付）

- 悬浮窗输入（Android Accessibility Service）
- 系统分享 sheet 集成（Soul / 相册截图直接分享进来）
- 微信平台适配（日常使用层；导入层 P0 已支持）
- 复盘报告（B 方向：聊完后整段对话复盘与改进建议）
- 多设备 / 多账号画像合并

## Phase 3（P2）

- 虚拟人预演（C 方向：基于自己历史聊天，AI 扮演典型人设供低风险预演）
- 反向筛选器（D 方向：分析对方对你哪一面有反应，识别真实匹配信号）
- 副驾 + 反应分析合体（E 方向）
- PDF 聊天技巧学习
- 小红书 / 陌陌等平台适配
- 本地小模型选项（隐私优先用户）

## Key Assumptions to Validate（按风险排序）

- [ ] 🔴 **海量历史数据中"我"的部分能被准确分离**（说话人识别在跨设备、跨时间、含群聊的场景下错位率 < 2%）
- [ ] 🔴 **采样策略能避免画像偏置**（验证：用不同对象的样本子集分别训练，画像核心维度应稳定，不被某段密集对话主导）
- [ ] 🔴 **批量数据能提取稳定可泛化的性格画像**（自评命中率 > 75%，比小样本目标更高）
- [ ] 🟡 **OCR + 说话人识别在 Soul 截图上准确率够高**（错位率 < 5%）
- [ ] 🟡 **批量 LLM 调用的成本可控**（DeepSeek 处理几万条消息的预算估算需实测）
- [ ] 🟡 **切 App 操作的体验损耗用户能接受**（自用 1 周后还想用）
- [ ] 🟡 **候选生成 P95 延迟 < 3 秒**
- [ ] 🟡 **3 候选 vs 1+refresh 哪种交互更好**（自用观察）
- [ ] 🟡 **采纳率作为优化信号是否有效**（事后自评一致性 > 70%）
- [ ] 🟢 **DeepSeek 的中文聊天风格模仿质量满足要求**（如不行则切 Claude）

## Open Questions

- 风格画像具体 schema 长什么样？需要原型 schema（语言风格、情感、幽默类型、回避模式、节奏的字段定义）
- 微信 PC 端导出的格式如何解析？是否依赖 WeChatMsg 等第三方工具？
- 批量导入数百张截图时，OCR 是本地（PaddleOCR）还是云端（成本/隐私权衡）？
- 数据清洗规则如何配置化？不同来源（微信/Soul/纯文本）噪声模式不同
- 采样策略具体算法？是按时间窗口分桶，还是按对话对象分桶，还是混合？
- 风格指纹版本升级时旧版本是否保留？支持回滚？
- 截图分享 sheet 集成的最佳交互是什么？
- 批量导入完成后是否给用户展示画像可读化总结（"你是个会冷幽默的人，回避情感话题"），让用户校准/确认？
